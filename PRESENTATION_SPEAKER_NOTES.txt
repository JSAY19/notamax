СЛАЙД 1. ТИТУЛ / ТЕМА ПРОЕКТА
--------------------------------------------------------------------------------

Наш проект называется notamax. Это веб-приложение — мессенджер прямо в браузере,
как обычный сайт, но с упором на приватность.

Ключевое слово, которое будет звучать дальше, — E2E, end-to-end encryption,
по-русски часто говорят «сквозное шифрование». Что это значит простыми словами:
сообщение превращается в нечитаемый набор данных на устройстве отправителя
и снова становится читаемым только на устройстве получателя. Всё, что между
ними — например, наш сервер в интернете — не должно уметь прочитать текст.
Это принципиально отличается от ситуации, когда сайт шифрует трафик до сервера
(HTTPS), а на сервере сообщение уже в открытом виде: у нас сервер выступает
только «почтальоном» для уже зашифрованного конверта.

Стек технологий на слайде — это инструменты: React и TypeScript на стороне
интерфейса, Node.js на стороне сервера, и канал связи в реальном времени
через WebSocket (о нём подробно на отдельном слайде).


--------------------------------------------------------------------------------
СЛАЙД 2. ЗАДАЧА: ЧТО МЫ ХОТЕЛИ СДЕЛАТЬ
--------------------------------------------------------------------------------

Задача была такая: два человека в браузере могут переписываться так, будто
через чат, но без того чтобы какой-то центральный сервер видел содержимое
переписки. Сервер нужен только чтобы «доставить байты» от клиента А к клиенту Б.

Мы специально не делали постоянное хранение истории на сервере в базе данных:
в нашем учебном проекте история сообщений живёт в памяти вкладки браузера.
Как только пользователь закрыл вкладку или обновил страницу — локальная
история в этом сеансе пропадает. Это сознательное упрощение и одновременно
идея «шпионского режима»: не остаётся копии переписки на нашем бэкенде.

Отдельно мы добавили локальную «учётную запись» на устройстве: пользователь
придумывает имя, генерируется криптографическая пара ключей, а секретный
материал упаковывается и кладётся в localStorage браузера, защищённый ключом
восстановления, который показывается один раз. Это не логин через наш сервер —
сервер про этот ключ ничего не знает. Плюс список друзей тоже хранится локально:
мы обмениваемся ключами шифрования только с теми именами, которые пользователь
сам внёс в список, чтобы не принимать шифротекст от случайных ников.


--------------------------------------------------------------------------------
СЛАЙД 3. ОБЩАЯ АРХИТЕКТУРА (КТО С КЕМ СОЕДИНЯЕТСЯ)
--------------------------------------------------------------------------------

Архитектура очень простая: два клиента — это два открытых у пользователей
браузера, и один сервер на Node.js. Каждый клиент открывает постоянное
соединение с сервером через технологию WebSocket (в нашем коде это библиотека
Socket.io, которая как раз даёт удобные «события» поверх сокета).

По этому соединению бегают два типа полезной нагрузки. Первый тип — публичные
ключи для криптографии: это не секрет, их можно показывать кому угодно, они
нужны, чтобы вычислить общий секрет между двумя сторонами. Второй тип —
уже зашифрованные сообщения: набор байтов, который без секретного ключа
превратить обратно в текст нельзя.

Сервер не вызывает функции расшифровки. Он только смотрит: «это сообщение
адресовано пользователю с таким именем» — находит его сокет и пересылает
туда тот же JSON с полями вроде идентификатора, от кого, кому, вектора
инициализации и шифротекста. То есть сервер — ретранслятор (relay), а не
«умный» обработчик текста.


--------------------------------------------------------------------------------
СЛАЙД 4. ФРОНТЕНД: ИНТЕРФЕЙС И СОСТОЯНИЕ
--------------------------------------------------------------------------------

На стороне пользователя мы используем React — это библиотека для построения
интерфейса из компонентов: список друзей, окно чата, поле ввода. TypeScript
добавляет статическую типизацию: мы описываем структуры данных для сообщений,
для пользователей онлайн, для событий сокета — меньше ошибок при разработке.

Стили — Tailwind CSS: утилитарные классы в разметке, быстро собирается тёмная
тема и внешний вид «как в мессенджере».

Состояние приложения — кто вошёл, список друзей, карта «общий секретный ключ
для каждого собеседника», карта сообщений — хранится в Zustand. Это лёгкая
библиотека для глобального состояния, по смыслу как упрощённый Redux: один
стор, подписка компонентов на нужные куски данных. Важно для нашей темы:
сообщения лежат в оперативной памяти процесса вкладки браузера, пока вкладка
жива. Отсюда формулировка «чат только в RAM»: RAM — это оперативная память
компьютера, куда браузер кладёт текущие объекты JavaScript. Это не диск:
после закрытия вкладки эти структуры не сохраняются в наш проект автоматически
как архив переписки на сервере.

Сборку и dev-сервер для фронтенда делает Vite — быстрый перезапуск при правках.


--------------------------------------------------------------------------------
СЛАЙД 5. БЭКЕНД И СОКЕТЫ: ЧТО ЭТО И КАК РАБОТАЕТ
--------------------------------------------------------------------------------

Обычный сайт по протоколу HTTP работает по принципу «запрос — ответ»: браузер
спросил страницу, сервер отдал и соединение часто закрывается. Для чата в
реальном времени так неудобно: пришлось бы каждые полсекунды опрашивать сервер,
есть ли новые сообщения.

WebSocket — это другое: после рукопожатия поверх HTTP открывается постоянный
двусторонний канал. Сервер может в любой момент отправить данные клиенту,
и клиент — серверу, без нового HTTP-запроса на каждое сообщение. Это похоже
на «трубу», по которой текут небольшие пакеты.

Socket.io — библиотека поверх WebSocket (с запасным вариантом через long polling,
если прокси режет WebSocket). В коде мы подписываемся на именованные события:
например, клиент отправляет join с именем пользователя, сервер рассылает
обновлённый список онлайн-пользователей, клиенты обмениваются public-key и
encrypted-message. Express у нас поднимает HTTP-сервер, к которому Socket.io
прикручен — типовая схема.

Итого: сокет — это долгоживущее соединение для push-уведомлений и чата;
наш сервер на Node только маршрутизирует события между сокетами по имени
получателя.


--------------------------------------------------------------------------------
СЛАЙД 6. ОБМЕН КЛЮЧАМИ: ECDH (БЕЗ ОБЩЕГО ПАРОЛЯ ПО СЕТИ)
--------------------------------------------------------------------------------

Чтобы шифровать сообщения симметричным алгоритмом (один ключ на двоих),
нужно, чтобы у обоих клиентов оказался один и тот же секрет, но передавать
этот секрет по сети нельзя — его перехватят.

Решение — протокол Диффи — Хеллмана на эллиптических кривых, ECDH. У каждого
участника генерируется пара ключей: секретный (никому не показываем) и
публичный (можно отправить по открытому каналу). Математика устроена так, что
если А знает свой секрет и публичный ключ Б, а Б знает свой секрет и публичный
ключ А, то оба независимо вычисляют одно и то же общее значение. Перехватчик,
который видел только публичные ключи, за разумное время это значение восстановить
не может при правильных параметрах.

В браузере мы используем Web Crypto API: кривая P-256 (NIST), стандартная для
интероперабельности. Публичные ключи сериализуем в формат JWK — JSON Web Key,
удобный для отправки в событии сокета.

После того как у обеих сторон есть общий секрет ECDH, мы из него выводим
именно ключ для AES — это делает функция deriveKey в том же API: на входе
наш секретный ключ ECDH и публичный ключ собеседника, на выходе — ключ
для симметричного шифрования сообщений.


--------------------------------------------------------------------------------
СЛАЙД 7. ШИФРОВАНИЕ СООБЩЕНИЙ: AES-256-GCM
--------------------------------------------------------------------------------

AES — Advanced Encryption Standard, стандарт симметричного блочного шифрования.
Симметричный значит: один и тот же ключ и шифрует, и расшифровывает. 256 — длина
ключа в битах; чем длиннее ключ, тем больше перебор в лоб при атаке.

Режим GCM — Galois/Counter Mode. Простыми словами: это не «голый AES», а режим,
который даёт два свойства сразу. Первое — конфиденциальность: из шифротекста
без ключа не узнать текст. Второе — целостность (аутентификация): к сообщению
прилагается специальный тег; если кто-то подменил хоть байт в передаче,
расшифровка на стороне получателя завершится ошибкой. То есть мы защищаемся
не только от чтения, но и от тихой подмены сообщения.

Для каждого нового сообщения генерируется случайный IV (initialization vector),
у нас 12 байт. IV не секретный и может передаваться открыто вместе с шифротекстом,
но для GCM критично не использовать один и тот же IV с одним и тем же ключом
для разных сообщений — поэтому каждый раз новый случайный IV.

На сервер уходит, по сути: кому, от кого, IV в Base64, шифротекст в Base64.
Получатель берёт свой сохранённый общий ключ с отправителем, IV и ciphertext,
вызывает decrypt в Web Crypto — получает исходную строку UTF-8.


--------------------------------------------------------------------------------
СЛАЙД 8. КАК СООБЩЕНИЕ ИДЁТ ОТ ОДНОГО ПОЛЬЗОВАТЕЛЯ К ДРУГОМУ (ПОШАГОВО)
--------------------------------------------------------------------------------

Шаг 1. Оба пользователя вошли в чат, сокет подключён, сервер знает их имена
и привязку «имя — id сокета».

Шаг 2. Оба добавили друг друга в список друзей (имена должны совпадать с тем,
что введено у другого, иначе клиент игнорирует чужие ключи и сообщения).

Шаг 3. Клиенты обмениваются публичными ключами ECDH через событие public-key:
сервер находит сокет получателя по полю «кому» и пересылает payload.

Шаг 4. Каждый клиент вычисляет общий секрет с другом и держит в памяти AES-ключ
для этого контакта.

Шаг 5. Пользователь вводит текст и жмёт «Отправить». В браузере вызывается
encrypt: случайный IV, AES-GCM, получается ciphertext. Формируется объект
с id сообщения, from, to, iv, ciphertext, опционально таймер самоуничтожения.

Шаг 6. Через сокет отправляется encrypted-message. Сервер пересылает только
адресату.

Шаг 7. У получателя срабатывает обработчик: если ключ уже есть — сразу decrypt
и добавление в список сообщений в Zustand; если ключ ещё не успел прийти,
сообщение можно поставить в очередь до завершения обмена ключами.

Шаг 8. Отправитель локально сразу показывает своё сообщение в интерфейсе
(он знает plaintext), а получатель после успешной расшифровки может отправить
короткое подтверждение message-delivered для галочек «доставлено» в UI.

Ни на одном из этих шагов сервер не вызывает crypto.subtle.decrypt — у него
нет ключей.


--------------------------------------------------------------------------------
СЛАЙД 9. ДОПОЛНИТЕЛЬНЫЕ ФУНКЦИИ И «СПЯЩИЙ» РЕЖИМ
--------------------------------------------------------------------------------

Самоуничтожение сообщений: таймер задаётся на клиенте; по истечении времени
сообщение удаляется из локального стора. Сервер не хранит таймер как сущность
в базе — это чисто поведение интерфейса и памяти.

Стеганография в проекте — демонстрационная: зашифрованная или исходная строка
кодируется в невидимые Unicode-символы внутри «обычного» текста. Для наблюдателя
виден безобидный фрагмент, а получатель может извлечь скрытый слой. Это не
дополнительная криптостойкость, а обфускация вида сообщения.

QR-код для публичного ключа — чтобы визуально передать или сверить JWK с другом
вживую (как идея проверки идентичности, без гарантии от подделки экрана).

Про «только в RAM» ещё раз: оперативная память вкладки — это быстрое хранилище
текущего сеанса. Мы не пишем ленту чата на наш сервер в файлы или БД. После
F5 история в приложении пустая (кроме того, что пользователь мог сохранить
сам). Локально в браузере отдельно лежат зашифрованный профиль и список друзей
в localStorage — это уже диск браузера, но не наша серверная база.


--------------------------------------------------------------------------------
СЛАЙД 10. ИТОГИ, ОГРАНИЧЕНИЯ, ЧТО МОЖНО УЛУЧШИТЬ
--------------------------------------------------------------------------------

Итог: мы собрали сквозное шифрование end-to-end на реальных браузерных API,
с сервером-посредником без доступа к plaintext, с обменом ключами ECDH и
сообщениями в AES-256-GCM, с интерфейсом на React и доставкой в реальном
времени через WebSocket/Socket.io.

Ограничения честные: имена пользователей на сервере не гарантируют глобальную
уникальность без отдельной регистрации; нет формального аудита протокола
как в продакшен-мессенджерах; перенос истории между устройствами не заложен
как облачная фича — только локальный зашифрованный профиль на одном браузере.

Направления развития: уникальные идентификаторы пользователей на сервере,
экспорт/импорт identity-файла для другого устройства, групповые чаты
с более сложной криптографией, и отдельно — любые UX-улучшения.

Если коротко для аудитории: мы показали, как классическая схема «ключи у клиентов,
сервер только пересылает байты» выглядит в коде на TypeScript и как к ней
подключаются современные веб-технологии.
